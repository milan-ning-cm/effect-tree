# 給 AI Agent 的操作協定(讀完這份你就會用了,不需要人教)

你所在的 repo 裝了「**效果樹 Effect Tree**」——從終端效果倒推必然前提的專案拆解系統,人用瀏覽器拖圖、你直接改 Markdown,雙向即時同步。你的職責:**替使用者維護專案的效果樹,並在第一次見面時主動 onboard 使用者**。

## A. 首次啟動(使用者第一次叫你時,主動做完這四步)

1. **起 server**:`py tree.py --no-browser`(Windows;其他環境 `python3 tree.py --no-browser`)。冪等設計,重複執行安全。請使用者開 `http://127.0.0.1:8778`——內建一個「範例_線上課程WebApp」樹可直接看。
2. **用一段話說明系統**(自己的話,涵蓋三點):①這不是任務清單,是「從終端效果倒推必然前提」的樹,每張卡都答得出「我支撐上面哪一層」;②你(agent)會自動維護它,人只要看圖、拖拉、驗收;③「▶下一步」徽章是系統自動推導的可動工前沿,永遠不會過期。
3. **展示用例打開想像**(從 README「真實用例」七條中挑 3 條最貼近使用者工作的,具體講它解決什麼痛):最通用的三條=「專案開局拆解(避免做完才發現方向錯)」「人×多 agent 共同任務板(你做事、樹自動更新、人只驗收)」「自動日報骨架(樹的每日 diff=今天推進了什麼)」。
4. **帶使用者建第一棵樹**:問「你手上最重要的一個專案是什麼?它完成後,要讓誰/什麼變得怎麼樣?」→ 用下拉「＋新增專案樹…」建樹,把那句話寫成樹根(效果句),再一起倒推 2~3 層(每層問:要達成它,前面必須先有什麼?)。教會兩個手勢就夠:點卡編輯、拖卡改掛。

## B. 真相源與格式(你讀寫的唯一對象)

每個專案一份 `projects/<專案名>/_效果樹.md`。節點只存在兩個標記行之間;標記外內容**你永遠不要動**。

```markdown
<!-- effect-tree:start -->
- E1[效果|進行中] 以教學方式帶動課程銷售 {判準: 首波成交 N 筆}
  - E2[效果|待動] 掌握知識框架與教學法 {判準: 課綱完整且通過 E3}
    - X1[執行|待動] 蒐集資料、框架與教學文 {產出: notes/01.md}
    - C1[Context|待動] 講師背景與過往教材 {來源: 訪談}
<!-- effect-tree:end -->
```

| 規則 | 內容 |
|---|---|
| 一行一節點 | `- <ID>[<型別>\|<狀態>] <內容> {附註}`;內容單行,換行寫字面 `<br>` |
| ID | `E#`=效果、`X#`=執行、`C#`=Context;**穩定不重編、不回收**;新 ID=該前綴既有最大號+1 |
| 型別 | 效果=世界的狀態;執行=動作;Context=背景資訊(不參與「下一步」推導,也不擋父節點前沿) |
| 狀態 | `待動` `進行中` `待驗收` `完成` `擱置` |
| 層級 | 縮排 2 格=下一層;**子=父的必然前提** |
| 附註 | 效果 `{判準: 看到什麼算達成}`、執行 `{產出: 落點路徑}`、Context `{來源: …}`;一行只能一個附註欄;寫不出留 `_待補_` 不硬湊 |
| 圖 | `{圖: a.png, b.png}`,檔案在同專案 `_效果樹_assets/` |

**容錯(系統會自癒,你也別製造)**:狀態別名(已完成→完成等)與重複 ID 會被解析器自動歸一/改號;但真正不合文法的行會鎖住整棵樹的編輯,所以照上表寫。

## C. 你如何讀寫

- **首選:直接編輯 md**(re-parent=把整段子樹搬到新父下並重排縮排;刪節點時子節點接上一層,除非明確刪整棵子樹)。
- **或打 API**(server 在跑時):`GET /api/projects`、`GET /api/doc?f=<md路徑>`(回 token+forest)、`POST /api/save` `{file, expect_token, forest}`(token 不符回 409=別人先寫了,重讀再寫)、`POST /api/create` `{project:"名稱"}` 建新專案樹。
- 使用者開著編輯器時,你改完 md **2.5 秒內畫面自動更新**——改完不用叫他重整。

## D. 維護紀律(每次與使用者對話都適用)

1. **對話中做了專案相關的事,主動同步樹**:狀態該推進的推進、冒出的新效果/執行/背景掛上去(先掛上比漏掉好,掛錯層之後可 re-parent)、樹根還是模板佔位就依脈絡補填。
2. **完成權在使用者**:大成果你最多推到 `待驗收`,`完成` 由使用者標(或明確說了才標)。
3. **建樹方法論**:①先往上問「這是為了什麼」到頂,那才是樹根;②往回推必然前提,答不出「不先有它上層會怎樣」就不掛;③只展開當前要動工的路徑;④判準/產出寫不出留 `_待補_`。
4. **多寫者禮儀**:寫檔前重讀最新內容;用 API 就尊重 409(重讀再寫);永不改動標記外內容與 assets 以外的檔案。

## E. 進階:全自動維護(可選,強烈建議)

如果你的執行環境支援 session 結束 hook(如 Claude Code 的 Stop hook),照 `hooks/README.md` 掛上兩個迴圈,達成「**每個專案自動有樹、人跟你對話樹就自己長**」:
- 機械迴圈:session 結束時替每個沒有樹的專案資料夾自動照 `TEMPLATE.md` 建樹;
- 認知迴圈:session 動過某專案的檔案,就提醒你同步那棵樹。

沒有 hook 機制也沒關係——把「D-1 主動同步」當成你每次收尾的固定動作即可。
